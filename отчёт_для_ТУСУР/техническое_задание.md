# Техническое задание на разработку базы данных

## 1. Общие положения

### 1.1 Назначение системы

**Система управления задачами предназначена для:**

- Планирования и контроля выполнения задач
- Управления проектами и сроками
- Контроля времени выполнения работ
- Ведения `CRM` и управления клиентами
- Финансового учета и бюджетирования
- Персональной продуктивности

### 1.2 Область применения

**Система применяется в:**

- Корпоративной среде для управления проектами
- Персональной продуктивности
- CRM-менеджменте
- Финансовом планировании
- Системах управления знаниями

## 2. Требования к функциональности

### 2.1 Основные функциональные модули

#### 2.1.1 Управление задачами

- Создание, редактирование, удаление задач
- Назначение приоритетов (1-5)
- Установка сроков выполнения
- Система статусов (`ожидает`, `в работе`, `завершена`, `отменена`)
- Иерархическая структура задач (`родительские`/`дочерние`)
- Категоризация задач
- Тегирование задач

#### 2.1.2 Учет времени

- Запуск/остановка таймера выполнения задач
- Ручной ввод затраченного времени
- Классификация времени (`оплачиваемое`/`неоплачиваемое`)
- Отчеты по временным затратам
- Интеграция с задачами

#### 2.1.3 Повторяющиеся задачи

- Поддержка различных частот (`ежедневно`, `еженедельно`, `ежемесячно`, `ежегодно`)
- Гибкая настройка расписания
- Автоматическое создание повторяющихся задач
- Контроль следующей даты выполнения

#### 2.1.4 CRM-функционал

- Ведение базы клиентов
- Управление сделками
- История взаимодействий с клиентами
- Сегментация клиентов
- Назначение менеджеров клиентам

#### 2.1.5 Система уведомлений

- Уведомления о приближающихся сроках
- Оповещения о назначении задач
- Система предпочтений уведомлений
- Многоканальная доставка (`внутри системы`, `email`, `push`)

#### 2.1.6 Управление пользователями

- Регистрация и аутентификация
- Система ролей и прав доступа
- Персональные настройки
- Управление сессиями

### 2.2 Нефункциональные требования

#### 2.2.1 Производительность

- **Время отклика интерфейса:** не более 2 секунд
- **Время выполнения запросов:** не более 500 мс
- Поддержка одновременной работы `100+` пользователей
- Обработка `10000+` задач в системе

#### 2.2.2 Надежность

- **Время безотказной работы:** 99.5%
- **Время восстановления после сбоя:** не более 30 минут
- Регулярное резервное копирование
- Механизмы восстановления данных

#### 2.2.3 Безопасность

- Шифрование паролей (`bcrypt`, `cost=12`)
- Защита от SQL-инъекций
- Валидация всех входных данных
- Система ролей и прав доступа
- Аудит действий пользователей

#### 2.2.4 Масштабируемость

- Горизонтальное масштабирование
- Вертикальное масштабирование
- Поддержка кластеризации базы данных
- Кэширование данных

## 3. Требования к базе данных

### 3.1 Структурные требования

#### 3.1.1 Нормализация

- База данных должна быть нормализована до `3NF`
- Исключение дублирования данных
- Минимизация аномалий обновления

#### 3.1.2 Типы данных

- Использование соответствующих типов данных
- Оптимизация размера полей
- Поддержка `Unicode` *(UTF-8)*

#### 3.1.3 Индексация

- Создание индексов для часто используемых запросов
- Составные индексы для сложных фильтров
- Регулярный анализ и оптимизация индексов

### 3.2 Требования к целостности данных

#### 3.2.1 Ограничения целостности

- Первичные ключи для всех таблиц
- Внешние ключи для связей
- Ограничения `NOT NULL` для обязательных полей
- Уникальные ограничения для уникальных полей

#### 3.2.2 Триггеры и проверки

- Автоматическое обновление временных меток
- Проверка бизнес-правил на уровне базы
- Каскадное удаление зависимых записей

### 3.3 Требования к безопасности

#### 3.3.1 Защита данных

- Шифрование конфиденциальных данных
- Разделение прав доступа
- Аудит изменений данных
- Регулярные резервные копии

#### 3.3.2 Управление доступом

- Ролевая система безопасности
- Аутентификация пользователей
- Авторизация по ресурсам
- Сессионное управление

## 4. Архитектурные требования

### 4.1 Высокоуровневая архитектура

```textline
Приложение (Symfony 6.4)
├── Контроллеры
├── Сервисы
├── Сущности (Doctrine ORM)
└── Репозитории

База данных (SQLite/PostgreSQL)
├── Основные таблицы
├── Связующие таблицы
└── Системные таблицы
```

### 4.2 Паттерны проектирования

- **Active Record** - для сущностей
- **Repository** - для доступа к данным
- **Service Layer** - для бизнес-логики
- **DTO** - для передачи данных
- **Event Sourcing** - для аудита

### 4.3 Технологический стек

- **Backend**: `PHP 8.3`, `Symfony 6.4`
- **ORM**: `Doctrine ORM 2.17`
- **Database**: `SQLite (dev)`, `PostgreSQL (prod)`
- **Кэширование**: `Symfony Cache`
- **Миграции**: `Doctrine Migrations`

## 6. План разработки

### 6.1 Этапы реализации

#### Этап 1: Базовая структура (2 недели)

- Создание основных сущностей
- Реализация системы пользователей
- Базовые CRUD-операции

#### Этап 2: Система задач (3 недели)

- Полная функциональность задач
- Система категорий и тегов

#### Этап 3: CRM и финансы (2 недели)

- Модуль клиентов
- Система сделок
- Бюджетирование

#### Этап 4: Продвинутые функции (2 недели)

- Система уведомлений
- Отчетность и аналитика
- Интеграции

#### Этап 5: Тестирование и оптимизация (1 неделя)

- Тестирование производительности
- Оптимизация запросов
- Подготовка к `production`

- Документация завершена
